## 2026-07-04 — Primer código de producto: arquitectura multi-proyecto + capa de datos (EF Core + SQLite)

Primer código real del proyecto. Arrancada la tarea #1 del Finde 1. Cambios de arquitectura y datos (racional
completo en `context/decisions.md`, 3 decisiones nuevas del 04-07):

- **Solución multi-proyecto:** creada `DididaiApp.sln` con `DididaiApp` (web, presentación) +
  `DididaiApp.Core` (class library: dominio, datos, servicios). Web referencia a Core. Convención del usuario.
- **Modelo de datos:** `Socio` (persona) disociado de `Colaboracion` (aportación) en 1:N. `Colaboracion` es
  jerarquía **TPH** (`CuotaDomiciliada`/`AportacionUnica`/`Teaming`), modelada sobre el formulario real de
  la web. `Importe` unificado en la clase base (evita 3 columnas que generaba EF).
- **EF Core 10 + SQLite:** `AppDbContext` en Core, registrado en `Program.cs`, connection string en
  `appsettings.json`. `dotnet-ef` actualizado 9.0.8 → 10.0.9. Migración `InitialCreate` aplicada → `dididai.db`
  creada. `.gitignore`: añadido `*.db` (RGPD, no subir datos reales).
- **Seguridad:** detectada vulnerabilidad transitiva `NU1903`/CVE-2025-6965 en `SQLitePCLRaw.lib.e_sqlite3
  2.1.11` (vía EF.Sqlite). Analizada: no explotable en nuestra superficie (sin SQL arbitrario) y sin parche
  disponible aún. Decisión: documentar y aceptar sin silenciar el aviso; revisar antes del deploy.

Decisiones de diseño acordadas con el usuario durante la sesión (cuestionó y corrigió): disociar pagos de
socio, herencia TPH, `Importe` único, y mobile-first como principio de UI (aún sin tocar front).

## 2026-07-04 — Autenticación con ASP.NET Core Identity (tarea #3 del Finde 1)

Montada la auth del back cerrado, manteniendo el front público abierto. Identity con Default UI y roles sobre
el `AppDbContext` (migración `AddIdentity`). Login, logout y recuperación de contraseña (con `IEmailSender`
stub que loguea en vez de enviar). **Registro público deshabilitado** por middleware (404 salvo Admin) tras
descubrir que la vía por convención de página no bloqueaba la Default UI. Zona `/Admin` protegida por rol;
seed de rol Admin + usuario admin con credenciales en User Secrets (nunca en el repo). Racional completo en
`context/decisions.md` (04-07).

**Verificado de verdad end-to-end** (peticiones HTTP, no solo compilar): la verificación destapó dos bugs que
se corrigieron sobre la marcha — faltaba `.AddDefaultUI()` (páginas de Identity daban 404) y las rutas de
cookie apuntaban a `/Account/Login` en vez de al área Identity. Registro anónimo abierto detectado y cerrado.
Tras las correcciones, todos los escenarios pasan (público abierto, /Admin protegido, login OK, registro solo
Admin). BD recreada limpia para eliminar un usuario de prueba.
