## 2026-07-14 — Bloque 4: log de auditoría transversal

Última pieza de código acordada del MVP. Registro automático e inmutable de las acciones de gestión, con
página de consulta de solo lectura. **No** sustituye a `AccionSolicitud` (gestión manual por solicitud): esto
es traza automática transversal a todos los módulos.

**Alcance (decidido con el usuario: "ampliado"):** 13 tipos de acción — alta/edición/baja/reactivación de
socio, alta/edición/baja de colaboración, aprobar/cancelar/vincular solicitud, alta/desactivación/reactivación
de admin.

**Diseño (decisión del usuario: traza EN LAS PÁGINAS):** el `RegistrarAsync` lo llama cada página tras la
acción exitosa, pasando `User.Identity?.Name`. Se sigue el patrón ya establecido en `AccionSolicitud`
(usuario como string inyectado por el llamante) y se mantiene Core **sin dependencia de HTTP** — ningún
`IHttpContextAccessor` en la capa de dominio. Alternativa (registrar dentro de los servicios) descartada:
obligaría a propagar `usuario` a todas las firmas y a que Core conociera la sesión. Precio asumido: hay que
acordarse de llamar a la traza al añadir acciones nuevas; mitigado dejando el patrón visible.

**Piezas:**
- Entidad `RegistroAuditoria` (Fecha UTC, Usuario, Accion enum, Entidad, EntidadId como **string** —ids no
  homogéneos: int para socios/colab/solicitud, GUID para admins—, Detalle legible ≤500).
- `IAuditoriaService`/`AuditoriaService`: `RegistrarAsync` (solo-inserción, fecha la fija el servicio, trunca
  detalle, respalda usuario vacío a "desconocido") + `ListarAsync` con filtros (usuario parcial, acción, rango
  de fechas con "hasta" inclusivo por día), orden desc y **paginación** (la tabla crece sin baja lógica).
- Migración **aditiva** `AddRegistroAuditoria` (solo crea la tabla + índice por Fecha; no toca nada existente).
- Página `/Admin/Auditoria` (solo lectura): filtros GET + tabla + paginación, CSP-safe (0 inline), tema de
  marca. Enlace en el menú del back y card en el panel de gestión.
- 8 tests nuevos → **145 verdes**.

**Verificado E2E por HTTP** (Playwright sigue bloqueado por el entorno): login admin → alta de socio registra
`SocioAlta` con detalle y usuario correctos; baja de socio registra `SocioBaja`; listado ordenado desc con el
total; filtro por acción aísla el tipo; filtro por usuario inexistente da tabla vacía; CSP presente y 0 inline.
La edición se probó pero el POST HTTP falla por validación de teléfono (el form parte prefijo/número por JS que
el script no ejecuta) — es artefacto del test, no del código: la edición usa el mismo `RegistrarAsync` en el
caso de éxito ya ejercitado por alta y baja. Datos de prueba borrados de la BD local al cerrar.

**Nota de proceso:** varios falsos positivos iniciales en la verificación HTTP (DNI duplicado con el socio
preexistente; regex sobre el HTML del form re-renderizado). Lección: verificar contra la BD/redirect real, no
contra el contenido del HTML devuelto por un POST que puede ser el form con errores.

**Limitación asumida:** la traza va tras la acción y en su propia transacción; si fallara la escritura de
auditoría (muy improbable, misma BD local) la excepción propaga en vez de silenciarse — preferible en un TFM.

**Pendiente:** commit + push + deploy; traducir EN del front; entregables no-código (README/slides/vídeo).

## 2026-07-14 — Auditoría: diff antes/después en las ediciones (feedback del usuario)

El usuario probó el Bloque 4 y señaló que "editó el socio" no dice **qué** cambió ni el valor previo. Se añade
el detalle campo-a-campo de las ediciones. Decisiones (con el usuario): diff **en el servicio** (excepción
justificada a "traza en las páginas": el estado "antes" solo está limpio dentro de `ActualizarAsync`, que ya
carga la entidad original tracked por EF), alcance **socio + colaboración**, formato **columna JSON nueva**.

- **Helper `ConstructorCambios`** (Core/Models): acumula `Registrar(campo, antes, después)` solo si difieren;
  `ToJson()` → JSON `{"Campo":{"antes":..,"despues":..}}` o `null` si nada cambió.
- **Columna `RegistroAuditoria.Cambios`** (string?, nullable) + migración aditiva `AddCambiosAuditoria`
  (solo `ADD COLUMN`).
- **`SocioService.ActualizarAsync`** y **`ColaboracionService.ActualizarAsync`** cambian su retorno a un record
  (`ResultadoActualizacionSocio` / `ResultadoActualizacionColaboracion`) que envuelve el enum + el JSON de
  cambios. Se actualizaron los llamantes (páginas Edit) y los tests afectados.
- **IBAN enmascarado** en el diff (solo últimos 4, resto `•`): no se vuelca el dato bancario en claro al log.
- **`RegistrarAsync`** gana parámetro opcional `cambios`. La vista `/Admin/Auditoria` deserializa el JSON y
  muestra líneas «Campo: antes → después» (∅ para vacío) bajo el detalle; CSP-safe.
- **Tests:** diff captura solo lo cambiado, `null` si nada cambia, IBAN enmascarado en el diff. Cuidado con el
  identity map de EF en los tests: hay que pasar una instancia DESACOPLADA (no la tracked) para simular el
  model binding, o `actual` y el entrante serían el mismo objeto y el diff no vería nada. **147 verdes.**

**Verificado E2E por HTTP:** editar un socio (nombre X→XEditado, teléfono, localidad ∅→Madrid) registra
`SocioEdicion` con las tres líneas antes→después. Datos de prueba restaurados en la BD local.

**Pendiente (sin cambios):** commit + push + deploy de todo el Bloque 4 (base + diff); traducir EN; entregables.

## 2026-07-14 — Pulido post-revisión: email en auditoría de admin + filtros del back en color de marca

Dos ajustes tras probar el usuario:

1. **Auditoría de admin mostraba el GUID en vez del email** ("Desactivación del administrador #31bec645-…").
   El handler `OnPostDesactivar/Reactivar` de `Usuarios/Index` solo recibe el id (por `asp-route-id`), así que
   el detalle se armaba con el GUID. Fix: resolver el email por id (`ListarAdminsAsync`, con respaldo al id si
   no aparece) ANTES de la acción y usarlo en el detalle → «Desactivación del administrador «pepe@…»». Los
   registros ya escritos con GUID no se reescriben (log inmutable); solo los nuevos salen con email. Verificado
   E2E (crear admin de prueba → desactivar → detalle con email).

2. **Filtros del back salían azul plano** (queja del usuario en Solicitudes). Causa: `btn-primary` estaba
   tematizado a naranja pero `btn-outline-primary`/`btn-outline-secondary` NO → usaban el azul/gris de
   Bootstrap. Fix en `site.css`: `btn-outline-primary` en naranja de marca (borde naranja + texto tostado;
   activo = relleno naranja, igual que `.btn-primary`) y `btn-outline-secondary` en tinta cálida. Arregla el
   azul en TODO el back, no solo en Solicitudes. CSS estático (recarga, sin recompilar). Verificado por HTTP.
